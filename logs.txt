## Best handoff chain

### 1) backend

Start here. This change is backend-first, because the review unit and API contract must change before the frontend can be honest.

**Why first:**  
The frontend cannot reliably implement page-by-page review while the backend still produces heading-derived section workspaces.

### 2) frontend

Once the backend exposes real page review units, switch the review UI from sections to pages.

**Why second:**  
This becomes mostly mapping and UX work once the contract is stable.

### 3) `Testing & QA`

Validate with a real uploaded PDF, not fixtures, and confirm:
- one review item per page
- evidence matches the page
- progress math is correct
- review completion advances correctly

### 4) `documentation` *(optional but recommended)*

Update architecture/status/docs after the behavior is confirmed.

## Recommended handoff prompts

### Prompt for backend

Please analyze and implement a backend-first change to convert manual document review from section-based units to page-based units.

Context:
- Current validation artifacts are already page-based in `*_validation.json` under `page_validations[]`
- Current review generation is section-based:
  - hitl_pipeline.py Stage 4 calls `extract_sections_from_markdown(...)`
  - section_review.py splits markdown on `##` headings
- Current API `GET /api/v1/documents/{document_id}/sections` is misleading:
  - it reads section files from the review workspace
  - parses `<!-- Lines: x-y -->`
  - returns those line numbers as `pageRange`, which is not true PDF page data

Goal:
- Make manual review page-by-page instead of section-by-section
- Use real PDF pages as the canonical review units
- Reuse existing per-page validation evidence instead of inventing section ranges later

Requirements:
1. Introduce a page-based review artifact and API contract
   - Prefer a new endpoint such as `/api/v1/documents/{document_id}/pages` or `/review-pages`
   - Do not break the existing `/sections` endpoint immediately unless clearly justified
2. Each review page should expose at minimum:
   - stable id
   - page number
   - status
   - markdown/extracted content for that page
   - text preview
   - validation issues for that page
   - evidence image paths / thumbnails
   - checklist state
3. Refactor review workspace generation so it no longer depends on markdown heading splits for the primary review flow
4. Preserve later stages where section grouping may still be useful, but treat sections as derived metadata rather than the manual review unit
5. Fix the current line-number/page-range mismatch
6. Add or update tests for:
   - page review artifact generation
   - page review endpoint
   - status/progress behavior
   - compatibility expectations if old section endpoint remains

Constraints:
- Keep scope focused on core document ingestion/review flow
- Keep auth-disabled local testing behavior intact
- Prefer backward-compatible rollout
- Root-cause fix, not UI-only patching

Please return:
- exact files changed
- new/updated API contract
- migration notes for the frontend agent
- known limitations if page markdown content is imperfect for some PDFs

### Prompt for frontend

Please update the review UI to support page-by-page manual review instead of section-based review.

Context:
- Backend is being changed to expose real page-based review units
- Today the review UI is built around `DocumentSection` and labels like “Sections reviewed”
- The current screen in Client.tsx expects:
  - `heading`
  - `content`
  - `pageRange`
  - checklist data
- The existing UX looks page-aware but is actually driven by section-derived data

Goal:
- Present one review item per PDF page
- Make the reviewer move through pages sequentially
- Align evidence, preview, checklist, and progress with actual page numbers

Requirements:
1. Replace section-centric wording and logic with page-centric wording:
   - “Sections reviewed” → “Pages reviewed”
   - list rail should show page numbers, not pseudo-sections
2. Consume the new backend page-based endpoint
3. For each page review item, display:
   - page number
   - extracted markdown/text for that page
   - evidence thumbnail(s)
   - validation issues tied to that page
   - checklist state
4. Keep the UX simple and reviewer-friendly:
   - prev/next page navigation
   - progress bar by reviewed pages
   - clear status badges per page
5. Do not reintroduce fixture-backed review behavior for active flows
6. Preserve:
   - `/PlantIQ` base path behavior
   - auth-disabled local manual testing mode
7. If a page has poor extracted markdown, show a graceful fallback with evidence and text preview instead of breaking the screen

Please return:
- files changed
- UI assumptions about backend contract
- any follow-up backend gaps discovered during implementation

### Prompt for `Testing & QA`

Please validate the new page-based review flow end-to-end using a real uploaded PDF.

Context:
- Review flow is being changed from section-based to page-based
- Validation artifacts are per-page
- Frontend review screen should now reflect actual PDF pages

Test goals:
1. Confirm one review unit per page
2. Confirm page numbers shown in UI match backend data
3. Confirm evidence images/thumbnails correspond to the correct page
4. Confirm checklist completion updates progress correctly
5. Confirm review completion advances document status as expected
6. Confirm no leftover line-number-as-page-range behavior remains
7. Test with at least one real uploaded document, not only fixtures

Please return:
- pass/fail summary
- exact defects found
- whether behavior is production-ready for core flow
- any regression risks in validation, review completion, or QA stages

### Optional prompt for `documentation`

Please update project status and relevant docs to reflect the shift from section-based manual review to page-based manual review.

Include:
- why the change was made
- old vs new review unit model
- API contract changes
- migration/deprecation notes for old section review artifacts/endpoints

## My recommendation on execution order

If you want the cleanest path, use this exact sequence:

1. backend
2. frontend
3. `Testing & QA`
4. `documentation`

I would **not** start with frontend here.  
That would just produce a prettier version of the same mismatch.

## A small but important design note

The best long-term model is:

- **Validation:** page-based
- **Manual review:** page-based
- **QA aggregation/reporting:** section/document-level
- **RAG chunking:** separate downstream concern

That separation keeps the reviewer workflow intuitive while still letting the system organize content semantically later.
